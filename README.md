# mcp-migrate

`mcp-migrate` analyzes an MCP server, over stdio or Streamable HTTP, and reports what needs to change for the MCP `2026-07-28` spec. Output is designed for automation: JSONL by default, or Markdown for human review.

> **Status**
>
> Pre-release targeting the MCP `2026-07-28` release candidate.
> The RC was locked on May 21, 2026, and the final spec is expected on July 28, 2026. Migration rules may change as the RC evolves and during final-spec reconciliation.
>
> The project is versioned `0.x`: the CLI interface and JSONL schema may still break.

## What Works Today

- Live `analyze` for Streamable HTTP.
- Live `analyze` for stdio.
- Read-only probes by default.
- JSONL output, one finding per line.
- Markdown rendering from the same finding model.
- Secret redaction in report output.
- Hidden-state detection for repeated live list probes.
- `patch` for safe mechanical source-code fixes (Go, JS/TS, Python).

Roadmap, not implemented yet:

- `watch` mode.
- Multi-language static scanning.

## Install

From the repo:

```sh
go build -o ./bin/mcp-migrate ./cmd/mcp-migrate
```

During development:

```sh
go run ./cmd/mcp-migrate help
```

## Quick Start

Analyze a Streamable HTTP server:

```sh
mcp-migrate analyze --transport http --url http://localhost:3000/mcp
```

Analyze a stdio server:

```sh
mcp-migrate analyze --transport stdio --server-command "node ./server.js"
```

Filter JSONL output with `jq`:

```sh
mcp-migrate analyze --transport http --url http://localhost:3000/mcp \
  | jq -r 'select(.severity == "breaking") | [.rule, .severity, .message] | @tsv'
```

Render a Markdown report:

```sh
mcp-migrate analyze --transport http --url http://localhost:3000/mcp --format markdown > report.md
```

Dry-run a patch (preview the diff without writing):

```sh
mcp-migrate patch --path ./src
```

Apply the patch:

```sh
mcp-migrate patch --path ./src --write
```

## Safety Guarantees

### analyze

Probes are read-only by default:

- `server/discover`
- `tools/list`
- `resources/list`
- `prompts/list`

Hidden-state detection repeats the same read-only list probes and compares canonicalized results across probe scopes. It does not call tools.

`resources/read` is not executed by default because a real server may attach side effects to a read: consumption, mark-as-read behavior, remote fetching, or other application-specific behavior. To allow it explicitly:

```sh
mcp-migrate analyze --transport http --url http://localhost:3000/mcp --allow-resource-read
```

`mcp-migrate` does not send any mutating `tools/call` during current live analysis.

Secrets are not written to JSONL or Markdown output: tokens, auth headers, environment variables, stderr, response bodies, raw headers, cookies, URL userinfo, and sensitive query parameters are redacted or omitted.

### patch

`patch` is dry-run by default: it prints a unified diff and reports skipped occurrences, but **writes nothing** unless `--write` is passed explicitly.

Substitutions are context-confirmed: `-32002` is only replaced when a resource-not-found signal (`resources/read`, `resource not found`, `resource_not_found`, or `ResourceNotFound`) appears within 12 lines of the occurrence. Occurrences without that signal are counted as `Skipped` and never touched.

Patches are idempotent: running `patch --write` twice produces no change on the second pass.

## Severity Legend

This legend is copied from the JSONL schema:

- `breaking`: incompatible with a strict MCP `2026-07-28` peer. This does not mean the feature stops working on July 28, 2026.
- `deprecated`: still functional in MCP `2026-07-28`, but in the Deprecated lifecycle state. Deprecated features remain functional for at least 12 months before earliest removal eligibility.
- `warning`: operational risk or minor non-conformance that may affect portability, scaling, or future migration.
- `info`: informational modernization or interoperability suggestion.

In practice, `breaking` means incompatible with a strict `2026-07-28` peer, not "breaks on July 28". `deprecated` features remain functional for at least 12 months before earliest removal eligibility.

## Report Model

JSONL output uses the `mcp-migrate/finding/v1` schema. Each line is an independent JSON object:

```json
{"schema":"mcp-migrate/finding/v1","rule":"server-discover-required","sep":{"id":"SEP-2575","status":"Accepted","verification":"unverified","source":"https://github.com/modelcontextprotocol/modelcontextprotocol/blob/main/seps/2575-stateless-mcp.md"},"severity":"breaking","enforcement":"report-only","spec_target":"2026-07-28","source":{"mode":"live","ref":"http://localhost:3000/mcp"},"message":"Server does not expose the stateless server/discover RPC.","remediation":"Implement server/discover with supported versions, server capabilities, and server identity.","autofix":false,"status":"confirmed"}
```

Empty output is valid: it means no findings were emitted.

SEP references that are not `Final`, or whose SEP file was not found during verification, are marked `unverified` in the output. They are not authoritative until the final `2026-07-28` spec has been reconciled, and they emit `enforcement: "report-only"` by default.

See [`docs/REPORT_SCHEMA.md`](docs/REPORT_SCHEMA.md) for the full schema.

## Positioning

`mcp-migrate` complements the official [`modelcontextprotocol/conformance`](https://github.com/modelcontextprotocol/conformance) suite. Conformance mainly answers a pass/fail question; `mcp-migrate` focuses on migration: severity, remediation, live compatibility signals, and eventually hidden-state detection. It does not replace the official conformance suite.

[`Janix-ai/mcp-validator`](https://github.com/Janix-ai/mcp-validator) is another useful reference in the ecosystem; it covers earlier specs and is not centered on the `2026-07-28` stateless migration.

## Feedback And Contributions

Useful feedback at this stage:

- observed divergence between a rule and the RC spec;
- JSONL output that is hard to use with `jq`;
- a finding that is too severe or not actionable enough;
- a real server that exposes a migration case not covered by the fixtures.

Keep changes focused: rules must trace to [`docs/SPEC_RULES.md`](docs/SPEC_RULES.md), and every non-final SEP reference must remain `unverified`.
